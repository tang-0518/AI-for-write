# 战役一验证原型

## 目标

验证百万字全托管写作引擎的两个核心假设：

1. **持续规划假设**：写完一幕再规划下一幕，是否会导致逻辑断裂？
2. **伏笔账本假设**：系统能否自动埋设和回收伏笔？

## 测试规模

- **3 幕 × 10 章 = 30 章**
- **预计字数：约 6 万字**
- **预计耗时：30-60 分钟**

## 运行方式

### Windows
```bash
cd scripts
run_prototype.bat
```

### Linux/Mac
```bash
cd scripts
python prototype_continuous_planning.py
```

## 验证流程

```
阶段 1: 初始化第一幕
  └─ 根据故事前提生成第一幕的 10 个章节大纲

阶段 2: 生成第一幕（10 章）
  └─ 逐章生成，自动埋设伏笔

阶段 3: 持续规划第二幕
  └─ 根据第一幕的内容和待回收伏笔，生成第二幕大纲

阶段 4: 生成第二幕（10 章）
  └─ 逐章生成，回收第一幕的伏笔，埋设新伏笔

阶段 5: 持续规划第三幕
  └─ 根据前两幕的内容和待回收伏笔，生成第三幕大纲

阶段 6: 生成第三幕（10 章）
  └─ 逐章生成，回收前面的伏笔

阶段 7: 生成验证报告
  └─ 统计字数、伏笔回收率、逻辑连贯性
```

## 成功标准

### 持续规划验证通过
- ✅ 成功生成 3 幕 30 章
- ✅ 每一幕的规划能够承接前一幕的剧情
- ✅ 没有明显的逻辑断裂

### 伏笔账本验证通过
- ✅ 伏笔回收率 >= 60%
- ✅ 自动检测到至少 5 个伏笔
- ✅ 伏笔在合理的章节范围内被回收

## 输出文件

所有结果保存在 `data/prototype_results/` 目录：

- `prototype_report_YYYYMMDD_HHMMSS.json` - 详细验证报告
- `prototype_novel_YYYYMMDD_HHMMSS.txt` - 完整生成的小说

## 报告内容


## 如果验证失败

### 伏笔回收率 < 60%
**原因**：伏笔提取逻辑过于简单（只检测关键词）

**解决方案**：
1. 使用 LLM 提取伏笔（调用 StateExtractor）
2. 在生成 Prompt 中明确要求埋设伏笔
3. 在回收时强制要求 LLM 回应伏笔

### 逻辑断裂
**原因**：幕与幕之间的上下文传递不足

**解决方案**：
1. 增加前情摘要的详细程度
2. 在规划下一幕时，提供更多前一幕的细节
3. 引入"幕级记忆"机制
